The following document will outline all relevant test procedures that will be incorporated in the programming of this system. It will give a description of what parts of the system will be tested, how they will be tested including methods and procedures, and actual test data (some of this will be alluded to rather than printed out, as it can all be accessed via the website, and makes for more concise reading). Each of the tests that we carry out will be documented and numbered, so that they can be referred to easily in the test results. Each test will consist of information regarding how it will be carried out, pre and post conditions and at what stage in the project it is to be executed. The various tests will be divided into different categories depending on at what level they are to be carried out. Various levels of testing will include:

- High level testing: Includes tests regarding usability of the GUI. For example, how easy is it to navigate the site, can a user do all the things he/she want to do, does the user only have access to the parts of the system they should do and so on. Most of these tests will be done when the system is complete, or very near completion.

- Class testing: These will be done during coding for the most part, probably on completion of a particular class. Does the class produce the correct data when it is passed the correct parameters. How do the classes interact with each other, do they communicate in the way that they have been designed to?

- General code testing: An on-going test procedure carried out while coding, including methods such as System.out.println statements. This part of the testing will not be documented in the same way as the above tests for obvious reasons, but is being mentioned as part of this document as it will make up a large percentage of how we will get the system to run in the correct way.

- Algorithm testing: Parts of the coding that are more complex will require detailed algorithm design and analysis and where possible, dry runs and test rigs will be constructed to ensure correct functionality before they are hardcoded. Again, these cannot be documented here as we need to start coding before these become aparant to us, but is being mentioned as it will be neccessary as part of the testing.

The first section of tests will refer to the usability of the system, i.e. how the user will interact with the system including expected output.

Passenger View

Test No. Test Description Pre Conditions Post Conditions
1.01 Passenger submits a query requesting information on a journey. All relavent text boxes filled in, destination station, beginning station, date and time, and any via stations required. User should be redirected to another screen showing them a list of all reasonable possible routes they could take to get to where they want to go. They should be able to scroll through a list of possible routes and drill down on any one particular train to see more information on it. They should also be able to have the option to book tickets from here.

Test No. Test Description Pre Conditions Post Conditions
1.02 Passenger views specific train details regarding a paritcular journey A query must have already been made by a passenger as above requesting information on a particular journey. They must have seen a list of possible trains that are available to them, and selected one from the list by clicking on it. Details corresponding to the specified train must be shown to the user on screen. This will include the type of train, where and when it stops en-route to it's destination, what facilities it has on board, and also a link to be able to book tickets for that particular journey.

1.03 Passenger can output details of journey as a hard copy (i.e. print journey details) The passenger must have selected the correct journey to print from the list made available to them. Details of the journey are printed out correctly when the appropriate button(s) is(are) pressed.

Test No. Test Description Pre Conditions Post Conditions
1.04 Passenger can make a booking and reserve a seat on the train. The passenger must have selected the journey that he/she wishes to book tickets for. Also, any seat reservation made must update the train info so that the station master can see how many seats have been booked so far for that particular journey.

Aims

To be able to start design after this meeting.
Decide requirements, view by view, working out externals before internals.

Passenger View

The written specification was read out. The user enters a time and stations to connect to, and trains from that time should be displayed.
It was suggested that the output should show the 2 trains that go before that time, and the two trains that arrive after that time,
It was suggested that the output should load trains that arrive 5 hours either side of the time. These should only be displayed 4 or 5 at a time in a scroll bar.
It was suggested that the train times shown should give basic information about the train (eg depart and arrival time), and when the time is clicked by the mouse, the entire details of that train should be shown.
It was suggested that there should be a dummy booking system allowing the user to reserve a seat on the train selected, producing a default amount of 100 spaces and a default price determined by the distance travelled.
These suggestions were agreed upon.

The user interface was discussed.
It was suggested that the user should have a pull down menu to select the stations from. This would stop users entering meaningless information, but would be very large and complex if there were many stations entered.
It was suggested this could be implemented with form completion, so that when the user enters a letter, it would show stations starting with that letter.
It was suggested that the user could select stations from a map representation of the network.
It was decided that the suggestions would be implemented together. There would be a small representation of the map on the screen, and if the mouse pointer rested over it, it would enlarge so that stations could be selected.

It was suggested that the 'via' option should be repeated so that the user may select several stations to go 'via'.
It was pointed out that this may use up unnessesary space on the interface.
It was decided that the via option should bring up another via option when completed. But no more than 3 vias should be allowed, otherwise the document becomes too complex.

It was suggested that there should be help for the user,
It was suggested that a message should be received by them if they try to enter 'stupid' data (ie asking to go by the same station more than once)
These suggestions were agreed upon.

Network Representation

The creation of the graphical network representation was discussed.
It was noted that the track data may not be as shown in the examples, so we could not produce a graph from this.
It was suggested that an algorithm could produce the network, however it was noted that the network produced would not have the stations in a correct position geographically to one another.
It was suggested that after the data is entered for the first time, the network operator should be able to move the stations to their 'correct' positions.
It was decided that the programme should draw the nodes and lines between them, and the network operator would be able to drag these nodes around, and the lines would follow.
It was decided that an algorithm to do this needed to be found.

Station Manager

The written specification was read out. The user enters a time and station and the programme outputs the next train to arrive at that station.
It was suggested that the manager should input a range of times, so that he could see the trains arriving in that period
It was suggested that this document should be produced as html, so that it could be printed and distributed to the employees at that station.
These suggestions were agreed upon.

It was suggested that if a booking system was implemented, that the driver should be able to see how many seats should be occupied on a particular train.
This was agreed upon, and it was also decided that he should be able to access the passenger view.

Network Operator

The written specification was read out. The network operator should be able to see what trains are on the network (graphically) at a specified time.
It was decided that we should assume that he has reasonabl intelligence and is geographically aware, which should be specified in the documentation.
As previously discussed, he should be able to change the representation of the map if the track data is new.
It was suggested that the graph produced should show arrows in the direction of the trains movement to represent the train. The markers should also display a number next to them which would represent the time to the next station.
It was suggested that the marker should be able to be clicked to produce information about that train.
These were agreed upon.

It was suggested that the trains should be moving in real time from when the system is initiallised. Alternatively, it was suggested that the view should be stationary, with an update button to see the system at a later point.
It was decided that the moving implementation would be used. This was suggested that it should have play, pause, fast forward and rewind buttons. These were also agreed upon.
It was decided that play should always show the system in real time from the initial time entered. (ie if it is paused for 10 minutes, pressing play would move the screen 10 minutes from that point). The fast forward and rewind buttons would allow the driver to skip to other times easily.
It was decided that the program should initially load in the network operator view, and he should be able to see the passenger and station manager views.

Other Business

The programme we should use to draw UML was discussed. Rhapsody was suggested but it was noted that this came with a license and would need to be paid for. It was decided that we should use ArgolUML which is freeware.
It was decided that we should have the specification written as soon as possible. This would allow us to draw up a plan for work to do over the holidays.
It was decided that everyone needed to practise java and UML over the holidays so we would be sharp for the next term.
It was also decided that testing would take at least a week, so we should aim to finish the coding with at least this amount of time to spare.
It was decided that those who had not set-up their accounts on event-horizon should do so directly after the meeting.

The meeting ended at 2:20pm.
